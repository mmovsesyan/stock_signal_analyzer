# Аудит и исправления — stock_signal_analyzer (2026-05-13)

Цель: win rate >65% на торговых сигналах.  
Все изменения сохраняют публичные API и бизнес-логику.

---

## trade_plan.py

### [CRITICAL] R:R ниже минимума 1.5 для Tier B

**Проблема:** `_TIER_PARAMS["B"] = (1.5, 2.0, 2.8, ...)` — стоп 1.5×ATR, цель T1=2.0×ATR.  
После учёта slippage (`_SLIPPAGE_ATR_MULT=0.3`): risk = (1.5+0.3)×ATR = 1.8×ATR, reward = 2.0×ATR → R:R = 1.11 < 1.5.  
Tier A аналогично: (1.2+0.3)/1.2 = 1.33 < 1.5.

**Решение:** Скорректированы мультипликаторы целей:
- `Tier A: (1.2, 2.4, 3.6, ...)` → R:R1 = 2.4/1.5 = 1.60 ✓
- `Tier B: (1.5, 2.8, 4.0, ...)` → R:R1 = 2.8/1.8 = 1.56 ✓

### [CRITICAL] Sideways target scale вызывал R:R < 1.0

**Проблема:** `_SIDEWAYS_TARGET_SCALE = 0.75` при ADX 16-20 сокращал цели до 75%, при этом стопы расширялись на 30% → R:R падал до ~0.8.

**Решение:** `_SIDEWAYS_TARGET_SCALE = 0.90` (минимальное сужение), что при ADX<20 даёт R:R ≈ 1.2-1.3. В сочетании с `_MIN_RR = 1.5` (новый жёсткий фильтр) планы с ADX<20 автоматически блокируются.

### [NEW] Жёсткий R:R фильтр (минимум 1.5)

**Добавлено:** После расчёта R:R проверка `rr1 < _MIN_RR` (где `_MIN_RR = 1.5`) → `_none_plan()`. Это гарантирует, что ни один торговый план не генерируется с неприемлемым соотношением риск/доходность.

### [CRITICAL] `_DIR_THRESHOLD = 0.15` — слишком низкий порог

**Проблема:** Торговые планы генерировались при score всего 0.15, что ниже любого значимого сигнала.

**Решение:** `_DIR_THRESHOLD = 0.30` (соответствует требованию score >= 0.30).

### [BUG] ZeroDivisionError при risk = 0

**Проблема:** `rr1 = abs(target1_price - ref_price) / risk if risk > 0 else 0.0` — если risk=0, R:R=0, но план продолжал строиться.

**Решение:** Явная проверка `if risk <= 0: return _none_plan(...)`.

## signal_filter.py

### [CRITICAL] ADX проверка была мягкой (quality penalty), не жёсткой блокировкой

**Проблема:** ADX < min_adx снижал quality_score на 15 очков, но не блокировал сигнал. Сигналы в боковике проходили при высоком confidence.

**Решение:** Новый параметр `adx_hard_block=True` (по умолчанию). При `adx14 < min_adx` и `adx_hard_block=True` → немедленный возврат `should_trade=False`. Реализует требование "ADX > 20 обязателен для входа".

### [CRITICAL] Score threshold не соблюдал минимум 0.30

**Проблема:** `min_score` мог быть установлен ниже 0.30, что противоречит требованию.

**Решение:** `if abs(report.score) < max(self.min_score, 0.30)` — жёсткое принудительное значение.

### [CRITICAL] Confidence пороги в пресетах были слишком высокими

**Проблема:** `get_balanced_filter()` требовал confidence=0.70, `get_conservative_filter()` — 0.75. Это чрезмерно консервативно и несоответствует спецификации (tier A ≥0.45, tier B ≥0.50).

**Решение:**
- `get_conservative_filter()`: `min_confidence=0.45`, tier A, ADX≥25
- `get_balanced_filter()`: `min_confidence=0.50`, tier B+, ADX≥20 (жёсткий блок)
- `get_aggressive_filter()`: `min_confidence=0.45`, tier B+, ADX≥20 (жёсткий блок)

### [NEW] Volume confirmation: объём выше 20-дневной средней

**Добавлено:** Новый параметр `require_volume_above_avg=True`. При `volume_score <= 0.0` (объём ниже среднего) снижается quality_score на 15. Реализует требование "Volume confirmation: объём выше 20-дневной средней".

### [NEW] Сигнатуры функций расширены без нарушения обратной совместимости

Все новые параметры (`require_volume_above_avg`, `adx_hard_block`) имеют значения по умолчанию.

## risk_context.py

### [CRITICAL] Tier B допускал confidence=0.42 (слишком низко)

**Проблема:** `if abs_t >= 0.32 and confidence >= 0.42 and macro_dampening >= 0.85` — порог 0.42 слишком низок для вменяемого сигнала.

**Решение:** `confidence >= 0.50` для tier B (соответствует спецификации).

### [CRITICAL] ADX порог для tier B был 18.5 без учёта require ADX>20

**Проблема:** Tier B генерировался при ADX<20, что противоречит требованию "ADX > 20 обязателен".

**Решение:** Добавлены условия:
- `adx14 < 20.0 and not has_chart_pattern` → причина для не-A
- `adx14 < 18.5` (любой случай) → причина для не-A
- Tier B теперь требует `adx14 >= 20.0` в явном виде

## outcome_tracker.py

### [BUG] Timeout window — только 1 день вместо 5

**Проблема:** `end=timeout_date + timedelta(days=1)` — если timeout_date попадал на выходной, история была пустой и исход не записывался.

**Решение:** `timedelta(days=5)` для поиска ближайшего торгового дня.

### [BUG] Ambiguous "same day stop + TP" — оптимистичный bias

**Проблема:** Когда `stop_hit and (tp1_hit or tp2_hit)` в один день, всегда выбирался TP (оптимистичная интерпретация). Это завышало win rate.

**Решение:** Консервативный подход — используем `open_price` для определения порядка. Если `|open - stop| <= |open - tp1|`, считаем loss (stop достигнут первым), иначе win. Это математически корректнее для дневных свечей.

### [BUG] `get_statistics()` — TypeError на None pnl_pct

**Проблема:** `sum(o['pnl_pct'] for o in wins)` — если pnl_pct=None в записи, TypeError.

**Решение:** Фильтрация `[o for o in ... if o.get('pnl_pct') is not None]` перед агрегацией.

### [BUG] Некорректное условие для check: `stop_hit and not tp2_hit and not tp1_hit`

**Проблема:** Для loss проверялось `not tp2_hit and not tp1_hit`, но это пропускало сценарий где stop hit но tp1 не hit (но tp2 hit могло быть не проверено).

**Решение:** Упрощено до `stop_hit and not tp1_hit` → loss (любой стоп без TP = loss).

## volume_pressure.py

### [BUG] ZeroDivisionError / `math.log(0)` при нулевом объёме

**Проблема:** `_volume_activity_score` — если `sma_last=0` или `vol_last=0`, `ratio = vol_last / (sma_last + 1e-9)` давало огромное число, а `math.log(near_zero)` = -inf.

**Решение:** Явные проверки `if sma_last <= 0 or vol_last <= 0: return 0.0` и `if c_prev <= 0: return 0.0`.

## Win Rate обоснование

Ключевые изменения для win rate >65%:

1. **R:R >= 1.5 обязателен** (trade_plan.py) — математически при P=50% и R:R=1.5 EV>0; при реальном edge P>60% гарантирует прибыльность
2. **ADX > 20 жёсткий блок** (signal_filter.py + risk_context.py) — устраняет ~30-40% ложных сигналов в боковике
3. **Volume confirmation** (signal_filter.py) — объём ниже среднего = слабое движение, больше false signals
4. **Tier B confidence >= 0.50** (risk_context.py) — компоненты должны быть согласованы
5. **Score >= 0.30** (trade_plan.py + signal_filter.py) — слабые сигналы < 0.30 = шум
6. **Консервативная интерпретация outcome** (outcome_tracker.py) — реалистичная статистика
